使用time命令

使用gprof

实验发现：gprof可以显示每个函数的调用次数和时间占比，帮助定位性能瓶颈。

使用perf

关键指标： - Cache miss rate：缓存未命中率，过高会影响性能 - Branch misprediction：分支预测失败率 - CPI：每条指令的周期数，理想情况下接近1

Cache机制

Cache层次结构

现代CPU通常有三级Cache： - L1 Cache：最快（1-2周期），最小（32-64KB） - L2 Cache：较快（10周期），中等（256KB-1MB） - L3 Cache：较慢（40周期），较大（几MB-几十MB） - 主存：最慢（200+周期），最大（GB级别）

时间局部性：最近访问的数据很可能再次被访问

空间局部性：与当前访问数据相邻的数据很可能被访问

流水线与分支预测

指令流水线

现代CPU使用流水线技术，将指令执行分为多个阶段：

  1. 取指（Fetch）
  2. 译码（Decode）
  3. 执行（Execute）
  4. 访存（Memory Access）
  5. 写回（Write Back）

流水线的挑战：分支指令会破坏流水线。

实验心得

1. 理解底层很重要

通过PA2实验，深刻理解了汇编指令、内存层次结构、流水线等底层知识。这些知识对写出高性能代码至关重要。

2. 性能优化是权衡

优化通常需要在多个方面权衡： - 时间与空间 - 代码复杂度与性能 - 可读性与执行效率

3. 测量是关键

不要凭感觉优化，一定要用工具测量： - time - gprof - perf - Cachegrind

4. 编译器很智能

现代编译器（gcc -O2/O3）能自动做很多优化： - 循环展开 - 函数内联 - 常量传播 - 死代码消除

所以，写清晰的代码比手动微优化更重要。

5. 瓶颈可能在意想不到的地方

通过实验发现，很多时候性能瓶颈不在算法复杂度，而在： - Cache未命中 - 分支预测失败 - 内存对齐 - 数据依赖

总结

PA2实验让我深入理解了程序在底层是如何执行的，以及如何优化程序性能。虽然现代编译器很智能，但理解底层原理仍然很重要，特别是在性能关键的场景。

实验过程中最大的收获是：性能优化不是玄学，而是有科学依据的。通过工具分析，找到真正的瓶颈，再有针对性地优化，才能事半功倍。
