记录kb-assistant中RAGAS评估框架集成、指标解读与Bad Case分析

这篇记录kb-assistant的性能评估系统，包括RAGAS框架集成、自定义指标、Bad Case分析等。评估是RAG系统迭代优化的关键。

为什么需要评估？

开发kb-assistant过程中发现： - 凭感觉优化不可靠 - 需要量化指标指导优化方向 - 需要发现系统的弱点 - 需要跟踪迭代效果

RAGAS框架

RAGAS是一个专门评估RAG系统的框架。

支持的指标

检索阶段

  • context_precision：检索结果中有多少是相关的
  • context_recall：相关文档被召回的比例
  • context_entity_recall：实体召回率

生成阶段

  • faithfulness：答案是否忠实于上下文（检测幻觉）
  • answer_relevancy：答案与问题的相关性
  • answer_correctness：答案的正确性
  • answer_similarity：答案与标准答案的相似度

端到端

  • answer_correctness：整体正确性

典型案例

案例1：检索召回率低

问题： - 问题："kb-assistant如何配置Qdrant？" - 预期答案：包含配置URL、端口、集合名称 - 实际答案：只提到用Docker启动Qdrant - Context Recall: 0.3

分析： - 查询词"配置Qdrant"被分词为["配置", "Qdrant"] - 相关文档中没有同时包含这两个词 - 向量检索找到了相关文档，但BM25未命中

解决： - 增加同义词：配置 → 设置/部署 - 提高向量检索权重 - 查询改写：将"如何配置"改为"Qdrant配置参数"

效果：Context Recall从0.3提升到0.7

案例2：答案幻觉

问题： - 问题："kb-assistant支持哪些Embedding模型？" - 上下文：只提到CLIP - 实际答案：声称支持OpenAI、BERT、CLIP等 - Faithfulness: 0.2

分析： - LLM根据训练知识回答了问题 - 但这些知识不在提供的上下文中 - 模型在"编造"额外的信息

解决：

效果：Faithfulness从0.2提升到0.85

案例3：答案不完整

问题： - 问题："评估RAG系统有哪些指标？" - 答案：只提到了faithfulness和context_recall - 实际应该包括：faithfulness、answer_relevancy、context_precision、context_recall、answer_correctness等 - Answer Completeness: 0.4

分析： - 检索到的文档只覆盖了部分指标 - 需要增加检索深度

解决： - 增加top_k从10到20 - 使用RRF融合提升排名靠后的相关文档 - 查询改写：添加"完整"、"所有"等词

效果：Answer Completeness从0.4提升到0.75

总结

评估的重要性： - 量化指标指导优化方向 - Bad Case分析发现具体问题 - 跟踪迭代效果

关键经验： - 不要只关注单一指标 - Faithfulness > 0.8是底线 - Context Recall > 0.7才合格 - 持续评估和优化

通过评估系统，kb-assistant的质量有了明显提升。特别是Faithfulness从0.71提升到0.85，说明答案的可信度大幅改善。

下一步： - 建立自动化评估流水线 - 增加用户反馈收集 - 实现在线学习 - 支持A/B测试
