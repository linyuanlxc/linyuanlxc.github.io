kb-assistant开发记录-概念篇

发表于 2025-08-15 分类于 项目开发 本文字数： 2.2k 阅读时长 ≈ 8 分钟
记录kb-assistant项目中关于RAG与Agent的知识整理与学习心得

这篇是kb-assistant开发记录的第一篇，主要整理RAG（检索增强生成）和Agent相关的概念。这些都是我在开发过程中学到的知识，记录下来方便回顾。

项目背景

kb-assistant是一个个人知识库助手，目标是： - 统一管理我的所有学习笔记、文档、代码 - 支持混合检索（文本、图片） - 提供流式问答 - 能够评估回答质量

RAG核心概念

什么是RAG？

RAG = Retrieval（检索） + Augmentation（增强） + Generation（生成）

简单说就是：先检索相关知识，然后把这些知识作为上下文提供给大语言模型，让模型基于这些知识回答问题。

工作流程：

为什么需要RAG？

  1. 解决幻觉问题：LLM不知道的知识会瞎编，RAG提供真实上下文
  2. 知识更新：不用重新训练模型，只需更新知识库
  3. 可解释性：可以查看答案的来源
  4. 领域知识：专业知识可以存入知识库

检索方式

在kb-assistant中，我实现了四种检索方式：

1. Dense Retrieval（向量检索）

  • 把文本转换为向量（embedding）
  • 计算向量相似度
  • 优点：语义理解能力强
  • 缺点：需要向量数据库，检索速度较慢

2. BM25（关键词检索）

  • 基于TF-IDF的改进算法
  • 计算关键词匹配度
  • 优点：速度快，精确匹配
  • 缺点：不理解语义

3. Graph Retrieval（图谱检索）

  • 从Neo4j图数据库中检索
  • 提取实体和关系
  • 优点：适合关系型查询
  • 缺点：需要维护图谱

4. Image Retrieval（图片检索）

  • 使用CLIP模型
  • 文本和图片在同一向量空间
  • 优点：支持多模态
  • 缺点：需要GPU加速

混合检索融合

在kb-assistant中，我没有只用一种检索方式，而是将四种结果融合：

为什么要融合？

  • 不同检索方式有各自的优缺点
  • 加权合并体现业务优先级
  • RRF补偿保证公平性
  • 实验结果：融合后效果比单一方式好

Agent概念

什么是Agent？

Agent = LLM + Memory（记忆） + Planning（规划） + Tools（工具）

简单说：让LLM能够使用工具、记住历史、进行规划的系统。

与传统RAG的区别： - RAG：一次检索，一次回答 - Agent：可以多次检索、使用工具、执行动作

Agent的工作流程

示例：用户问"今天北京的天气怎么样？"

  1. Agent理解意图：需要查询天气
  2. 调用天气API工具
  3. 获取天气数据
  4. 整理回答用户

提示工程（Prompt Engineering）

在kb-assistant中的实践

1. 查询改写

用户提问时，如果有多轮对话，需要改写查询使其独立：

示例： - 用户：它的原理是什么？ - 历史：前面在讨论RAG - 改写：RAG检索增强生成的原理是什么？

2. 答案生成提示

技巧： - 明确要求"优先依据上下文" - 告诉模型"不知道就说不知道" - 要求"给出简洁回答并附来源"

3. 少样本提示（Few-shot）

提示优化的经验

  1. 具体比模糊好：明确指出要做什么
  2. 例子很有帮助：给1-2个例子
  3. 角色设定有效：告诉模型"你是谁"
  4. 结构化输出：要求JSON格式或其他结构

评估指标

检索阶段

在开发kb-assistant时，需要评估检索质量：

  • Context Recall：检索内容是否包含了答案所需信息
  • Context Precision：检索结果中有多少是相关的
  • MRR：平均倒数排名，第一个相关结果的位置

生成阶段

  • Faithfulness：答案是否忠实于上下文（检测幻觉）
  • Answer Relevancy：答案与问题的相关性
  • Answer Correctness：答案的正确性

实现评估

开发中的思考

为什么用FastAPI而不是Flask？

  • FastAPI自动生成API文档
  • 类型提示更好
  • 异步支持更完善
  • 性能更高

为什么用Qdrant而不是Milvus？

  • Qdrant更轻量
  • API更简单
  • 部署方便（Docker一键启动）
  • 性能足够

为什么用Neo4j？

  • 知识图谱适合表示文档间关系
  • Cypher查询语言直观
  • 社区活跃
  • 可视化好

总结

通过kb-assistant项目，我深入理解了RAG的原理和实现。关键收获： - 混合检索比单一检索效果好 - 数据契约很重要 - 评估指标帮助定位问题 - 异步编程提升性能 - 工具链完善才能高效开发

整个开发过程中最大的体会是：不要过度设计。先实现MVP，再逐步优化。很多功能（如RRF融合）都是后期根据实际效果添加的。
