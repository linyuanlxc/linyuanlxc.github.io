记录kb-assistant混合检索架构设计思路与技术选型过程

这篇记录kb-assistant的架构设计过程，包括为什么采用FastAPI+Qdrant+Neo4j的技术栈，以及混合检索架构的设计思路。这些都是开发过程中的真实思考。

架构演进

最初的想法

最开始想做一个简单的知识库问答系统： - 用Flask做后端 - 用FAISS做向量检索 - 用OpenAI的Embedding API

问题： - Flask没有自动文档 - FAISS不支持增量更新 - OpenAI API有调用限制

第一次重构

改用： - FastAPI：自动生成文档，异步支持好 - Qdrant：支持增量更新，RESTful API - 本地CLIP模型：不受API限制

效果： - 部署方便了 - 支持增量索引 - 检索速度提升

第二次重构

发现单纯向量检索效果不够好，于是加入了： - BM25关键词检索 - Neo4j图谱检索 - 多路融合排序

效果： - 检索准确率提升30% - 能够回答关系型问题 - Bad Case减少

技术选型

为什么选FastAPI？

备选方案： - Flask：简单但功能少 - Django：太重，不适合API服务 - Tornado：异步但文档少

选择FastAPI的理由：

  1. 类型安全

  2. 自动文档

  3. 异步支持

  4. Pydantic模型

为什么选Qdrant？

备选方案： - FAISS：Facebook开源，性能好但功能少 - Milvus：功能强大但部署复杂 - Pinecone：云服务，有费用

选择Qdrant的理由：

  1. RESTful API

  2. 混合检索支持

  3. 增量更新

  4. 性能足够

  • 索引10000个文档 < 1秒
  • 检索延迟 < 100ms
  • 内存占用合理

为什么选Neo4j？

备选方案： - 不用图数据库：放弃关系检索 - SQLite：不是图数据库，查询复杂 - ArangoDB：多模型，但社区小

选择Neo4j的理由：

  1. Cypher查询语言直观

  2. 可视化好

  3. Python驱动成熟

  4. 适合知识图谱

为什么用CLIP而不是纯文本Embedding？

备选方案： - OpenAI text-embedding-3：效果好，但需要API - BERT：只能文本 - SentenceTransformer：多模型可选

选择CLIP的理由：

  1. 跨模态

  2. 本地运行

  3. 多语言支持

代价： - 模型较大（~300MB） - 需要GPU加速（否则速度慢） - 精度略低于专用文本模型

架构设计

整体架构

关键设计决策

1. 父子分块策略

为什么这样设计？

  • 子块小：检索更精确
  • 父块大：提供更多上下文
  • 关系明确：知道子块属于哪个父块

效果：回答的完整性和准确性都提升了。

2. 两阶段融合

为什么两阶段？

  • 加权合并：体现业务优先级（向量检索最重要）
  • RRF补偿：保证公平性（BM25排名靠前的也能得分）
  • 互补：既考虑分数，也考虑排名

3. 增量索引

优势： - 增量更新快（100个文档更新只需几秒） - 全量重建慢（10000个文档需要几分钟） - 节省计算资源

4. 错误降级

设计原则： - 核心功能必须可用（向量、BM25） - 增强功能可以降级（图谱、图片） - 记录日志方便排查

配置文件设计

三级配置

优点： - 灵活：开发用默认值，部署用环境变量 - 清晰：优先级明确 - 安全：敏感信息放环境变量

模型配置分离

好处： - 模型配置集中管理 - 更换模型只需改配置文件 - 支持多模型切换

总结

架构设计的关键决策： - FastAPI：现代、异步、自动文档 - Qdrant：轻量、RESTful、增量更新 - Neo4j：图查询直观、可视化好 - 混合检索：四路融合，效果好 - 错误降级：核心功能必须可用

架构演进的原则： - 从简单开始，逐步优化 - 监控驱动架构改进 - 保持模块低耦合 - 文档和代码同等重要

架构设计没有银弹，只有适合自己项目的才是最好的。kb-assistant的架构还在持续演进中，根据使用反馈不断调整优化。
