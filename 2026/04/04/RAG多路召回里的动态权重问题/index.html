RAG多路召回里的动态权重问题

发表于 2026-04-04 分类于 项目开发 本文字数： 3.3k 阅读时长 ≈ 12 分钟
从dense、BM25、graph、image四路召回出发，讨论为什么固定权重融合不够，以及更合理的查询自适应融合方案。

最近在 RAG 系统时，我把召回拆成了四路：dense、bm25、graph、image。最开始的融合方式很直接，就是给四路结果设固定权重，然后线性加权。这个方案能跑，但是这样是有问题的。比如用户问的明确是文本，这就不需要或者说要减小image的权重了。又比如用户问的是某个术语的准确定义、某段文本里的原话、某个 API 名称，这类查询显然更偏文本，bm25 和 dense 应该占主导，image 的权重通常应该很低。反过来，如果用户问“那张图里那个结构叫什么”或者“找一下包含某种界面的截图”，再把 image 当成一个固定的小配角就不对了。graph 也一样，它对关系型、多跳型、实体约束比较强的查询有价值，但不是每个 query 都值得走高权重。

固定权重的问题，本质上不是融合公式写得不够复杂，而是系统没有显式建模 query 和检索器之间的匹配关系。

固定权重的缺陷

固定权重隐含了三个假设：

  1. 不同检索器对所有 query 的贡献是稳定的。
  2. 不同模态、不同索引的分数经过归一化之后可以长期共用一套配比。
  3. 检索器的失误是均匀分布的，而不是 query-dependent 的。

这三个假设基本都站不住。

在实际系统里，query 的差异很大：

  • 有的是关键词精确匹配，BM25 天然占优。
  • 有的是语义改写，dense 更稳。
  • 有的是关系查询，graph 才是主场。
  • 有的是视觉指代或者图文对齐，image 才真正有用。

再往下看，哪怕同样是文本问句，也会分成“短 query、长 query、实体 query、组合约束 query、多跳 query、带视觉锚点 query”。这些 query 的最优融合权重，本来就不可能一样。

所以这里真正应该学的不是“怎么把四个分数加起来”，而是：

对当前这个 query，哪一路更该被相信？

这件事在近两年的做法里是怎么处理的

这类问题最近的思路已经比较清楚了，核心趋势不是继续调静态超参，而是把融合从“固定配方”改成“query-aware decision”。

1. 先做 query routing，再做融合

这一类方法会先判断 query 属于什么类型，再决定该走哪些检索路径、每条路径大概给多少权重。

HyPA-RAG 里已经很明确地把 query complexity classifier 放进检索环节，用复杂度分类来调 retrieval 参数，并在 hybrid retrieval 里组合 dense、sparse 和 knowledge graph。到了更近的多模态文档问答场景，AutoThinkRAG 进一步把 Query Complexity Router 单独拿出来，根据 query 的难度和结构选择不同的检索与推理路径。

这类工作的共同点是：先承认 query 之间是异质的，再让系统按 query 分流。

如果放到我这四路召回里，这一步至少应该回答两个问题：

  • 这个 query 主要是文本、关系、还是视觉驱动的？
  • 这个 query 是单跳检索，还是明显需要结构化约束和多跳扩展？

只要这一步做对一半，固定权重方案里很多明显的误召回就会少掉。

这一步的具体实现：让小模型判断查询意图，输出权重。这是 2024-2025 年业界最普遍的做法（LangChain、LlamaIndex、Dify 都有类似实现）。

但是这种方法其实还是在把决定权交给llm。对于一些明确的问题效果应该不错，但是对于用户描述混乱、存在前后逻辑矛盾的问题，可能效果就要差一些。在这一步之前起码是要做一个query优化改写的，甚至还要加上一个多维度改写，然后合并召回结果。

2. 用 query 本身预测融合权重

如果说 routing 更像“先分路再检索”，那另一条更直接的路线就是：直接从 query 表示预测融合权重。

今年的 Query-Adaptive Hybrid Search 就很典型。它的核心是一个 QDAP（Query-Driven Alpha Prediction） 模块，直接根据 query 的潜在表示去预测 dense / sparse 的最优混合系数，而不是给所有 query 共用一个静态 alpha。这类方法有个很实际的优点：延迟低，不需要在在线链路里再调用一个大模型做判断。

这条思路对四路召回同样成立。只不过二路的 alpha 会变成一个四维权重向量：

\[ w(q) = [w_{dense}, w_{bm25}, w_{graph}, w_{image}] \]

并且满足：

\[ \sum_i w_i(q)=1 \]

这里的重点不是公式，而是权重应该由 query 决定，而不是写死在配置里。

3. 不只看 query，还要看每一路检索结果本身的“置信度”。

这个思路我觉得很有参考价值，而且很有新意。

只看 query 还不够，因为有些 query 从字面上看偏文本，但实际某一路的返回结果已经明显在“胡来”了。这时候更合理的做法是把检索结果分布本身也当成信号。

[Entropy-Based Dynamic Hybrid Retrieval for Adaptive Query Weighting in RAG Pipelines]这篇工作走的就是这个方向：它不是只凭 query 来定权重，而是根据 sparse 和 dense 返回结果的熵、分布不确定性去做动态重加权。

这个思路我觉得很值得借到多路召回里。因为对工程系统来说，下面这些信号往往比“某个原始分数高了一点”更可靠：

  • top1 和 top2 的分差够不够大
  • top-k 分数分布是不是太平
  • 这一条路返回的结果是否高度重复
  • 不同路之间有没有交集
  • 某一路是否只召回出边角料，缺少主证据

如果 image 路的 top-k 很散、熵很高，而且和文本路几乎没有交叉证据，那它就不该还拿着固定的 0.2 权重硬往前挤。

我印象比较深的是论文 "Entropy-Based Dynamic Hybrid Retrieval"。他是用“各检索通道得分分布的熵”动态计算每一路的置信度，自动分配权重。 熵越低 → 结果越确定、越可信 → 权重越高。

熵（Entropy）：衡量结果分布的不确定性/混乱度 - 低熵：得分差距大、排名很“肯定” → 高置信 - 高熵：得分很平、谁都不突出 → 低置信

4. 在多模态场景里，关键不是“把 image 也接上”，而是“只在需要时接上”

多模态 RAG 的早期代表工作，比如 MuRAG，已经说明了图像和文本联合检索是有价值的。但更近一点的工作开始强调另一件更实际的事：视觉证据不是应该永远参加，而是应该按需参加。

SAM-RAG 的表述就很直接：它会根据输入 query 动态过滤相关文档，在需要时才引入 image captions。这和很多工程系统里“既然建了 image index，那每次都一起融”是两套思路。

这点其实很重要。因为图像检索的问题不只是慢一点，还包括它非常容易带来一种“看起来相关，但对答案没帮助”的噪声。尤其在文本主导的问题上，图像召回经常能提供一些视觉相似的上下文，但这些东西并不能真正支持答案，反而会污染后续 rerank 和生成。

重新设计

我现在更倾向于把它拆成三层，而不是继续调一组全局常数。

第一层：先做一个轻量的 query router

这一层不需要很重，甚至一开始不一定要训练大模型。目标只有一个：输出一个检索先验。

比如：

  • 文本精确匹配型：提高 bm25
  • 语义改写型：提高 dense
  • 实体关系 / 多跳型：提高 graph
  • 明确视觉指代型：提高 image

这里可以先从规则和小分类器起步。像“图里、截图、界面、这张图、看起来像”这类触发词，本来就是很强的视觉信号；而“关系、依赖、谁连接谁、上下游、调用链”这类说法，对 graph 也是明显加分项。

第二层：再用每一路返回结果做置信度修正

这一步比第一层更像“在线校准”。

我比较看重的不是原始 score，而是这些统计特征：

  • top-k score entropy
  • top1-top2 margin
  • source diversity
  • dense / bm25 / graph / image 之间的 overlap
  • 是否命中高价值字段，比如标题、实体名、caption、邻接关系

然后把第一层的 query 先验和第二层的检索置信度合起来，输出最终权重：

\[ w_i(q)=\text{softmax}(a \cdot p_i(q) + b \cdot c_i(q) - \lambda \cdot cost_i) \]

其中 p_i(q) 是 query router 给出的先验，c_i(q) 是检索结果置信度，cost_i 可以顺手把高延迟路径也纳入考虑。

这样做的好处是，系统既不会只凭 query 硬猜，也不会只看召回分数拍脑袋。

第三层：不要让“权重融合”承担全部排序责任

这一点非常重要。

动态权重能解决的是“不同检索器该占多大比重”，但它解决不了最终文档级排序的全部问题。更稳的做法还是在融合之后再接一层 reranker，最好是跨路统一排序。

换句话说：

  • 动态权重负责决定谁更值得相信
  • reranker 负责决定具体哪些候选真该进上下文

如果没有这第三层，前两层就算做得不错，也很容易因为 chunk 粒度、字段质量、图节点展开方式不同而留下排序噪声。

落地顺序

如果是工程落地，但我没做过工程落地，参考GPT大人的看法，不太建议一上来就追“端到端最优”。

更现实的顺序大概是：

  1. 先做一个规则版 router，把明显不该高权重的 image 和 graph 压下去。
  2. 离线标注一批 query，统计每一路单独召回和不同融合策略下的 Recall@k / nDCG / answer support。
  3. 用这些数据训练一个很轻的 gating 模型，比如 LightGBM / MLP，输入是 query 特征和各路 top-k 统计，输出四路权重。
  4. 最后再加统一 reranker。

这里最关键的是第 2 步。没有按 query slice 拆开的离线分析，动态权重很容易重新变成另一种拍脑袋。

我觉得至少要单独看这些切片：

  • 精确文本问答
  • 语义改写问答
  • 多跳关系问答
  • 图文混合问答
  • 明确视觉问答

只看 overall 平均分，很容易把问题看平。

个人理解

多路召回的重点，从来都不是“路越多越好”，而是系统是否知道什么时候该信哪一路。

固定权重融合的问题，不在于它太简单，而在于它把 query 的差异抹平了。近两年的做法其实已经给了比较清楚的方向：要么显式做 query routing，要么直接学习 query-aware 的融合权重，要么再进一步，把检索结果的不确定性也纳入权重预测。

如果只说一个结论，我现在会这么概括：

四路召回里真正应该学习的，不是一个全局最优权重，而是一个“按 query 动态分配检索信任”的机制。

这个机制未必要很重，但必须是 query-aware 的。否则路开得越多，噪声往往也越多。
