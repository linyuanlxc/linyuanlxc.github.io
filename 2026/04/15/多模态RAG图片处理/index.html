多模态RAG里图片到底该怎么处理

发表于 2026-04-15 分类于 项目开发 本文字数： 2.3k 阅读时长 ≈ 8 分钟
结合近年的多模态RAG、视觉检索和图表理解论文，讨论文档里的图片在RAG里到底该如何索引、检索和利用。

做多模态 RAG 时，一个很现实的问题是：原始文档里明明有图片，但检索阶段真的能把这些图片找出来吗？就算找出来了，大模型又是不是真的能稳定地读懂图里的信息？

这个问题不能只靠“把图片也塞进向量库”来解决。真正难的地方有两个：

  1. 图能不能被召回
  2. 图被召回后，能不能被可靠利用

这两件事其实是分开的。

文本 RAG的局限

传统 RAG 默认文档先 OCR / parse 成纯文本，再按 chunk 检索。这个流程对普通段落没问题，但对图文混排文档有天然损失。

VisRAG 讲得很直接：很多真实文档里的关键信息就在 layout、figure、page image 里，纯文本解析会把这些信息丢掉。ColPali 也是类似判断，它强调文档页本身就是视觉丰富的对象，不应该只被当成 OCR 后的文本。

这也是为什么很多系统会出现一种很典型的错误：

  • 文本 chunk 召回看起来没问题
  • 但真正支撑答案的证据其实在图里
  • 结果生成阶段只能“绕着图讲”，讲不准

再回到多模态。现在一个很现实的问题是,在RAG的原始文档中有图片,但是检索的时候能准确地检索到这些图片吗?就算检索到了这些图片,大模型能否有效地去提取图片中的信息? 现在我的一个想法是,在图片入库的时候,对图片的前后几个chunk,把这几个chunk做成embedding,然后与这个图片会形成一个映射,比如说把图片的chunkID记录在文字的元数据里面,然后把文字的chunk的ID记录在图片的元数据里面,这样实现一个映射。因为图片的前后几段的文字,大概率是会提到这个图片具体是干嘛的,有哪些内容。然后再针对一些数据图,比如说折线图这些,额外再增加一个小模型去理解这个图片的内容,识别这个图片中的数据,然后把识别到的数据存到元数据里面,这是我的一个想法。

图片召回问题

如果只说工程落地，其实不止上面那些，还可以做一些改进。我们可以参考知识图谱的思路，为什么不把图片和文本片段看作节点呢，图片节点和文本节点之间应该有关系，这个关系是可以作为显式边的，不能只靠 embedding 碰运气。

比较合理的做法是把文档拆成几类节点：

  • text chunk
  • figure / image crop
  • page
  • table / chart
  • section

然后建立几种关联：

  1. 邻近关系 图片前后几段文字、同页 chunk、同 section chunk。
  2. 引用关系 例如正文里出现“图 3”“Figure 2”“如下图所示”。
  3. 标题/说明关系 caption、图注、表注通常比前后段落更重要。
  4. 页面归属关系 图和页、chunk 和页之间的映射。

这样做以后，检索就不再是“只把图片当独立向量搜一下”，而是可以变成：

  • 文本命中后，顺着边把相关图片带出来
  • 图片命中后，顺着边把解释它的文字带出来

这比单纯把 chunk_id 塞进 metadata 更进一步。本质上已经接近 RAG-Anything 这类工作的思路了：把多模态内容看成互相连接的知识实体，而不是彼此孤立的数据块。

检索层我更倾向于“两套索引 + 一个关联图”

从最近几篇论文看，我觉得比较稳的架构不是单索引，而是三层：

  1. 文本索引

正常做 dense / BM25 / hybrid，这层负责大部分文本问题。

  1. 视觉索引

对页面图、图片裁剪、图表裁剪做视觉 embedding。这里如果是文档检索场景，近两年比较值得参考的是 ColPali 和 M3DocRAG 这一路：直接把页面当视觉对象来建检索，而不是先把它压成 OCR 文本。

  1. 关联图

这层不负责“语义相似检索”，而负责“证据扩展”。

比如用户问某张结构图的作用，视觉索引可能先找到图片；但真正适合塞进上下文的，往往是：

  • 图本身
  • 图注
  • 图前后的解释段落
  • 引用该图的正文

这些内容应该通过图结构扩出来，而不是依赖一次检索全猜对。

图片理解问题

这是第二个常被低估的问题。

通用多模态大模型对自然图片、界面截图、简单示意图通常还能看个大概，但一碰到数据图、复杂表格、密集标注图，稳定性就会明显下降，尤其是数值抽取。

看了一些论文，发现其实我想的把图喂给 VLM，然后由他来做信息提取是不太行的。

对图表类图片，更靠谱的路线是先做结构化中间表示。

DePlot 的核心想法就是把 chart 理解拆成两步：先把图转成表，再让语言模型基于表做推理。这个确实是个很好的思路，图片更像是文本的高维抽象表示，表格则是文本的被凝练后的关键数据事实。ChartInstruct 和 OneChart 也都说明了一件事：图表理解最好有专门模块，尤其数值和结构抽取不能完全指望通用 VLM。

所以之前说的“给折线图之类额外加一个小模型，把识别出的数据存进 metadata”，可以这样做，但是在这之前应该还有很多准备工作。

其实还可以考虑把图片增强拆成两类

第一类：普通图片

比如示意图、界面截图、照片、流程图。

这类图我会存这些字段：

  • page_id
  • figure_id
  • caption
  • nearby_chunk_ids
  • ref_chunk_ids
  • ocr_text（如果图里有字）
  • vision_embedding
  • text_surrogate

这里的 text_surrogate 很重要。它可以是图注、邻近段落摘要、VLM 生成的一段简短描述。作用不是替代图片，而是给文本检索一条“搭桥”的路。

第二类：数据图

比如折线图、柱状图、散点图、饼图。

这类图我不会只存 caption，而是尽量存结构化结果：

  • 图表类型
  • 坐标轴标题
  • 图例
  • 系列名
  • 抽取出的数据点或表格
  • 数值置信度

这样后面回答“峰值在哪”“哪一年开始下降”“A 是否高于 B”这类问题时，就不必每次在线让大模型重新数轴读图。

图片真的是必须的吗

多模态 RAG 的另一个误区是：既然图进库了，那每次都把图也塞给模型。

我更认同 SAM-RAG 这种思路：按需引入视觉证据。它的做法是根据 query 动态过滤文档，需要时才引入 image captions。这个想法很朴素，但很重要。

如果放到工程里，大概可以这么做：

  1. 先判断 query 是文本型、视觉型、还是图表型。
  2. 文本型优先走文本检索，再通过关联图补图片。
  3. 视觉型优先走视觉检索，再补图注和邻近段落。
  4. 图表型优先查图表结构化元数据，必要时再把原图送进 VLM 复核。

也就是说，VLM 更适合做后验核查和局部理解，而不是承担整个检索链路的主力。

推荐方案

综上，我觉得比较稳的多模态 RAG 方案大概是这样：

离线建库

  1. 解析 PDF / 文档版面，切出 page、chunk、figure、chart、table。
  2. 建文本索引。
  3. 建页面/图片视觉索引。
  4. 建 chunk <-> figure <-> page <-> section 的关联图。
  5. 对 chart 跑专门解析模型，抽结构化数据。
  6. 给每张图补一个轻量 text surrogate。

在线检索

  1. 先做 query 路由，判断是否需要视觉证据。
  2. 从文本索引、视觉索引分别召回。
  3. 通过关联图做证据扩展。
  4. 用 reranker 统一排序。
  5. 只把少量高价值图片交给 VLM。

在线生成

  1. 文本证据负责“讲清楚”
  2. 图片证据负责“补视觉事实”
  3. 图表结构化结果负责“保数值稳定”

这个方案的关键是把图片从“难处理的附件”变成“有结构的证据节点”。

一些个人理解

多模态 RAG 真正难的，从来不是能不能把图片也存起来，而是：

  • 检索时有没有办法把它找出来
  • 找出来之后有没有可靠的桥把它和文本证据接起来
  • 生成时有没有必要让模型重新看图，还是应该直接用已经抽好的结构化结果
